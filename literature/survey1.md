| 🏛️ Origin & team | 🏢 NVIDIA GEAR with UMich, Berkeley, CMU (2026) | 🏢 Google Research / Robotics team, Liang et al. (2022) | 🎓 Stanford, Li Fei-Fei's group, Wenlong Huang (2023) | 🏢 NVIDIA and UPenn, Yecheng Jason Ma (2023) |
| 📖 Venue | 📄 arXiv preprint, July 2026, not peer-reviewed yet | 🎤 Published at IEEE ICRA 2023 conference | 🎤 Conference on Robot Learning (CoRL) 2023 | 🎤 ICLR 2024 conference paper, widely cited |
| 📊 Citations (S2 · Jul '26) | 🆕 Zero citations — brand-new July 2026 release | 🔥 1,644 citations — the foundational, most-cited baseline | 📈 986 citations — highly influential value-map approach | 📈 655 citations — popular LLM reward-design method |

| 🎯 What the LLM writes | 🐍 Python control programs composing perception, planning, control | 🐍 Python policy code calling perception and control APIs | 🗺️ Code composing 3D affordance and constraint voxel maps | 🎁 Dense reward functions as executable Python code |
| 🧠 How it learns / improves | 🧬 Evolutionary program search plus iterative multimodal debugging | ⚡ No learning — single zero/few-shot code generation | ⚡ No learning — zero-shot map synthesis per instruction | 🧬 Evolutionary reward search, then 🎢 RL trains policy |
| 📉 Uses gradient training? | ❌ No gradients; skills stored as code, not weights | ❌ No gradient training; pure prompting at inference time | ❌ No gradient training; planning over generated value maps | ✅ Yes — reward code drives downstream RL gradient updates |
| 📚 Persistent skill library? | ✅ Yes — an ever-growing ♾️ library of reusable skills | ❌ No memory; each task is solved from scratch | ❌ No library; nothing is accumulated across tasks | ❌ No library; reward functions are not stored/reused |
| 🔁 Reuses past experience | ✅ Retrieves prior debugged fixes as in-context guidance | ❌ None — no cross-task knowledge carried forward | ❌ None — each instruction handled fully independently | ❌ None — search restarts for every new task |
| 👀 Failure feedback signal | 🎥 Per-primitive multimodal traces pinpoint exact failure cause | 🚫 No feedback loop; open-loop generated code just executes | 🔄 Closed-loop visual feedback used only for replanning | 📊 Reward reflection summarizes RL training dynamics/statistics |
| 🔍 Search strategy | 🧬 Proposes population of K candidate programs each round | 🚫 No search; one program is sampled directly | 🧮 No search; deterministic map-then-plan pipeline runs | 🧬 Parallel population sampling with reflection-guided reward mutation |
| 🕹️ Benchmarks / environments | 🕹️ LIBERO, Robosuite, BEHAVIOR-1K, plus real robots | 🖥️ Real tabletop manipulation and simulated drawing tasks | 🖥️ SAPIEN/RLBench simulation plus real Franka arm | 🕹️ Isaac Gym GPU simulation across 29 environments |
| 🔬 Real-robot validation | ✅ Yes — bimanual YAM station, a different embodiment | ✅ Yes — demonstrated on multiple real robot arms | ✅ Yes — real Franka Emika Panda experiments shown | ❌ No — simulation only (🐕 DrEureka later added real) |
| 🔢 Scale / coverage | 🌐 150+ tasks and 90+ self-taught reusable skills | 🖐️ Dozens of tabletop commands across robot platforms | ♾️ Open-set everyday tasks with open-set objects | 🎯 29 tasks, 10 morphologies, 🏅 beats humans 83% |
| 💾 "Trained model" artifact | 📦 A repository of human-readable sensorimotor skill code | 📄 A disposable script generated fresh per command | 📄 Disposable value maps discarded after each task | ⚖️ Standard neural RL policy weights (opaque tensors) |
| 🌉 Sim-to-real approach | 📤 Ships know-how; robot re-practices the strategy real | ➖ Not applicable; already runs on real hardware | ➖ Not applicable; deploys directly with no sim2real | ⚖️ Ships weights; needs domain randomization via DrEureka |
| 🦾 Cross-embodiment transfer | ✅ Skills transfer despite different arms and APIs | 🔧 Rewrite prompts and APIs for each new embodiment | 🔧 Re-specify perception and planner per new robot | 🔧 Retrain reward and policy for each morphology |
| ➕ Continual / open-ended? | ✅ Yes — indefinitely ♾️ compounding, open-ended skill growth | ❌ No — static, one-shot generation, no accumulation | ❌ No — static, nothing accumulates over time | ❌ No — isolated per-task optimization runs each time |
| 🧩 Beyond fixed primitive APIs? | ❌ No — limited to a predefined primitive API set | ❌ No — composes only the given control primitives | ❌ No — bounded by perception and planner primitives | ✅ Yes — invents dense low-level reward shaping signals |
| 📈 Scaling to many tasks | ✅ Cost amortizes; later tasks reuse earlier skills | ⚠️ Linear cost; a full rewrite for every task | ⚠️ Linear cost; fresh synthesis needed every task | ⚠️ Expensive; full reward search and RL retrain each |
| 💻 Open-source availability | ✅ Full stack open-source promised on public release | ✅ Yes — code and prompts were publicly released | ✅ Yes — GitHub repo with full implementation available | ✅ Yes — official repository under permissive MIT license |
| 💸 Compute / cost profile | 🔴 High — many LLM calls per evolutionary search loop | 🟢 Low — only a few LLM calls per command | 🟡 Moderate — LLM plus VLM queries per task | 🔴 High — many LLM calls plus GPU RL training |
| 🥇 Signature result / headline | 🤝 Bimanual handover success jumped from 20% to 92% | 🗣️→🐍 LLMs write deployable robot policies from language | 🚫📦 Zero-shot open-set manipulation from free-form language | 🖐️🖊️ Shadow Hand learned human-speed pen-spinning tricks first |
